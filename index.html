1.5 File Processing System

[cite_start]

The traditional computer-based method where each department in an organization has its own set of files, designed specifically for their own applications. [cite: 235, 237, 238] [cite_start]This system has many disadvantages. [cite: 248]

[cite_start]

Disadvantages of File Processing System: [cite: 253]

    [cite_start]
  • Data Redundancy and Inconsistency: The same data is duplicated in multiple files, which can lead to conflicts (e.g., an address is updated in one file but not another). [cite: 254, 258]
    • [cite_start]
  • Data Isolation: Data is stored in various files, making it very difficult to write new applications to retrieve data from multiple files. [cite: 262, 263, 264]
    • [cite_start]
  • Integrity Problems: It's hard to enforce consistency constraints (e.g., "Marks must be a number"). [cite: 267, 269, 271]
  • Program-Data Dependence: Programs are tied to the file format. [cite_start]If you change the file's structure (e.g., add a field), you must change all programs that use that file. [cite: 272, 273, 274]
    • [cite_start]
  • Atomicity Problem: A transaction (e.g., a money transfer) must happen entirely or not at all. [cite: 278] [cite_start]File systems can fail mid-step, leaving data inconsistent. [cite: 285, 286, 287]
    • [cite_start]
  • Security Problems: It's difficult to provide different levels of access (e.g., read-only vs. write) for different users. [cite: 289, 296]
    • [cite_start]
  • Program Maintenance: Programs are difficult to maintain. [cite: 300]

Chapter 2: The Database Environment

2.1 Three-Level Architecture

[cite_start]

The fundamental architecture for modern database systems, it separates the database into three levels or views to provide data independence. [cite: 1142, 2200]

  • 1. [cite_start]External Level / View: What a particular user or application sees. [cite: 1183, 1187] [cite_start]It's a "subschema" showing only the part of the database relevant to that user. [cite: 1197] [cite_start]There can be many external views. [cite: 1183]
  • 2. [cite_start]Logical or Conceptual Level / View: The "community view." [cite: 1202] [cite_start]A complete logical description of the *entire* database (all entities, attributes, relationships). [cite: 1200, 1201] [cite_start]It hides physical storage details. [cite: 1218] [cite_start]There is only one conceptual schema. [cite: 1232]
  • 3. [cite_start]Internal or Physical Level: Describes *how* the data is physically stored on the storage media (data structures, file organization, indexes). [cite: 1234, 1235, 1250] [cite_start]There is only one internal schema. [cite: 1250]

2.2 Mapping

[cite_start]

The process of converting or relating data between the three levels. [cite: 1252, 1253]

    [cite_start]
  • External/Conceptual Mapping: Defines the correspondence between a specific external view and the conceptual view. [cite: 1284]
    • [cite_start]
  • Conceptual/Internal Mapping: Defines the correspondence between the conceptual view and the stored database. [cite: 1257]

2.3 Data Independence

[cite_start]

The main advantage of the 3-Level Architecture. [cite: 1327] [cite_start]It means you can change the schema at one level without having to change the schema at the next level up. [cite: 1328]

    [cite_start]
  • Physical Data Independence: The ability to change the Internal level (e.g., add an index, change file organization) without affecting the Conceptual level. [cite: 1333, 1340] This is relatively easy to achieve.
    • [cite_start]
  • Logical Data Independence: The ability to change the Conceptual level (e.g., add a new entity or attribute) without affecting existing External views. [cite: 1345, 1348] This is much harder to achieve.

Chapter 3: The Entity-Relationship (E-R) Model

3.1 & 3.2 Elements of the E-R Model

[cite_start]

The E-R model (by Peter Chen, 1976) is a logical, visual representation of data. [cite: 2513, 2516] [cite_start]It's simple, provides a clear visual representation, and is an effective communication tool. [cite: 2519, 2523, 2525]

    [cite_start]
  • Entity: A person, place, thing, or event about which data is collected (e.g., STUDENT, BOOK). [cite: 2534, 2536]
      [cite_start]
    • Entity Type: A set of entities with the same attributes (e.g., the STUDENT class). [cite: 2546]
      • [cite_start]
    • Entity Instance: A single member of the set (e.g., the student 'Abdullah'). [cite: 2554, 2556]
      • [cite_start]
    • Entity Set: All entities of a particular type at a given time. [cite: 2559]
    • [cite_start]
  • Attribute: A characteristic or property of an entity (e.g., Name, Address). [cite: 2565]
      [cite_start]
    • Attribute Domain: The set of possible values for an attribute (e.g., GPA is 0-4). [cite: 2570, 2572]
    • [cite_start]
  • Relationship: A logical connection or association between entities (e.g., a STUDENT "Studies" a COURSE). [cite: 2577, 2580]

3.3.1 Types of Attributes

[cite_start]
Simple (Atomic): Cannot be subdivided (e.g., Gender). [cite: 2622]
[cite_start]
Composite: Can be subdivided (e.g., Address -> Street, City). [cite: 2628]
[cite_start]
Single-Valued: Has only one value (e.g., Age). [cite: 2640]
[cite_start]
Multi-Valued: Can have many values (e.g., Skills). [cite: 2642]
[cite_start]
Stored: Physically stored in the database (e.g., DateOfBirth). [cite: 2657]
[cite_start]
Derived: Calculated from a stored attribute (e.g., Age is derived from DateOfBirth). [cite: 2660, 2663]
[cite_start]Identifier (Key): An attribute that uniquely identifies one instance (e.g., StudentID). [cite: 2674, 2676]
  • Unique vs. Non-unique: `EmployeeID` is unique. [cite_start]`Department` is non-unique (identifies a set). [cite: 2686, 2687, 2688]
  • Simple vs. Composite: `Student_ID` is a simple identifier. [cite_start]`OrderID` (OrderNo + Date) is composite. [cite: 2690, 2692]

3.3.2 Types of Entities

  • Strong Entity: Can exist on its own and has its own unique identifier (e.g., PERSON). [cite_start]Represented by a single-line rectangle. [cite: 2720, 2721]
    • [cite_start]
  • Weak Entity: Cannot exist without its "owner" strong entity (e.g., CHILD depends on PERSON). [cite: 2712, 2733] [cite_start]Represented by a double-line rectangle. [cite: 2719]
  • Associative Entity: An entity that associates the instances of one or more entity types. [cite_start]Often used to "resolve" a many-to-many relationship and store attributes about that relationship (e.g., an `AWARD` links a `SCHOLAR` and `RESEARCH`). [cite: 2736, 2738, 2739]

3.4 Degree of Relationships

    [cite_start]
  • Unary (Recursive): A relationship between instances of the same entity type (e.g., a PERSON "Lives With" another PERSON). [cite: 2763, 2764, 2769]
  • Binary: A relationship between two entity types. [cite_start]This is the most common. [cite: 2774, 2757]
      [cite_start]
    • One-to-One (1:1): A CHAIRMAN "Runs" one DEPARTMENT. [cite: 2776, 2785]
      • [cite_start]
    • One-to-Many (1:N): A COLLEGE "Has" many DEPARTMENTS. [cite: 2799, 2808]
      • [cite_start]
    • Many-to-Many (M:N): A STUDENT "Studies" many COURSES. [cite: 2816, 2825]
    • [cite_start]
  • Ternary: A relationship between three entity types (e.g., a CONSULTANT, CLIENT, and CONTRACT are all linked by a "Signs" relationship). [cite: 2837, 2842]

3.4.4 Cardinality Constraints

[cite_start]

Specifies the number of instances of one entity that can be associated with an instance of another entity. [cite: 2851] It defines the "business rules."

    [cite_start]
  • Maximum Cardinality: The maximum number (1 or Many). [cite: 2876]
    • [cite_start]
  • Minimum Cardinality: The minimum number (0 or 1). [cite: 2873]
      [cite_start]
    • If minimum is 0, participation is Optional. [cite: 2874]
      • [cite_start]
    • If minimum is 1, participation is Mandatory. [cite: 2874]

3.5 Subtype & Supertype Entities

[cite_start]

A way to model IS-A relationships (e.g., "A STUDENT IS-A PERSON"). [cite: 2964] [cite_start]A supertype contains common attributes, while subtypes contain specific attributes and inherit from the supertype. [cite: 2919, 2987]

    [cite_start]
  • Supertype: The general entity (e.g., PERSON) that contains common attributes (Name, Address). [cite: 2961, 2963]
    • [cite_start]
  • Subtype: The specific entity (e.g., STUDENT) that contains specific attributes (RollNo) and inherits all attributes from the supertype. [cite: 2919, 2987]
    • [cite_start]
  • Generalization: The process of identifying a more general entity type (bottom-up). [cite: 2966]
    • [cite_start]
  • Specialization: The process of identifying more specific entity types (top-down). [cite: 2971]

Chapter 4: The Semantic Object Model

4.1 & 4.2 Semantic Object Model

[cite_start]

An alternative to the E-R model that views the system as a collection of objects. [cite: 3865, 3866] [cite_start]It's considered more closely aligned with a user's perception of the world. [cite: 3868]

[cite_start]

A Semantic Object is a "thing" in the user's environment, grouped into classes (e.g., STUDENT class). [cite: 3870, 3873] [cite_start]It has attributes: [cite: 3886]

    [cite_start]
  • Simple Attributes: Contain a single value. [cite: 3888]
    • [cite_start]
  • Group Attributes: Composites of other attributes (e.g., Address). [cite: 3890]
    • [cite_start]
  • Semantic Object Attributes: An attribute that *is* another semantic object (e.g., a COLLEGE object has a PROFESSOR attribute, which is a link to the PROFESSOR object). [cite: 3894, 3913]

4.2.2 Attribute Cardinality

[cite_start]

Specified in n.m format, where 'n' is the minimum (0 or 1) and 'm' is the maximum (1 or N). [cite: 3921, 3943, 3946] [cite_start]E.g., 1.1 (mandatory one), 0.N (optional many). [cite: 3947, 3948]

4.2.4 Object Identifiers

[cite_start]

An attribute or collection of attributes used to identify an object instance (e.g., Roll No). [cite: 3973] [cite_start]Can be unique (like a `Roll No`) or non-unique (like a `Name`). [cite: 3980]

4.3 Attribute Domains

A collection of all possible values for an attribute. A domain has two descriptions:

  • Physical Description: The data type (e.g., "string", "Roll No should not exceed 99").
  • Semantic Description: The function or purpose of the attribute (e.g., "Name of a student must be a valid name").

4.4 Semantic Object Views

A part of an object that is displayed in a particular application. One object can be displayed in many different views (e.g., one view for students, one for professors, both from the same COLLEGE object).

4.5 Types of Objects

  • Simple Object: Contains only single-valued, non-object attributes.
  • Composite Object: Contains one or more multi-valued, non-object attributes.
  • Compound Object: Contains at least one object attribute.
  • Hybrid Object: A combination of composite and compound.
  • Association Object: An object that links two or more other objects, typically storing data about that relationship.
  • Parent/Subtype Objects: A subtype object inherits the attributes of its parent (similar to E-R super/subtypes).
  • Archetype/Version Objects: An archetype object produces other objects that are "versions" of it (e.g., BOOK is the archetype, EDITION is the version).

[cite_start]

    • [cite_start]
  • Degree: The number of attributes (columns). [cite: 3996]
    • [cite_start]
  • Cardinality: The number of tuples (rows). [cite: 3996]

5.2 Keys

[cite_start]

Keys are attributes used to identify tuples (rows) in a relation. [cite: 3997]

    [cite_start]
  • Super Key: Any attribute or set of attributes that uniquely identifies a tuple. [cite: 3997]
    • [cite_start]
  • Candidate Key: A minimal super key (no attribute can be removed and it still be unique). [cite: 3997]
  • Primary Key (PK): The one candidate key chosen by the designer to be the main identifier. [cite_start]It cannot contain null values. [cite: 3997]
    • [cite_start]
  • Alternate Key: A candidate key that was *not* chosen to be the primary key. [cite: 3998]
    • [cite_start]
  • Composite Key: A primary key that consists of two or more attributes. [cite: 3998]
    • [cite_start]
  • Foreign Key (FK): An attribute in one relation whose values match the primary key in another relation, used to link the tables. [cite: 3998]

5.3 Relational Database Management System (RDBMS)

[cite_start]

A Relational Database Management System (RDBMS) is the software that manages relational databases. [cite: 3999] [cite_start]Its components include a File Manager, Database Manager, Query Processor, Data Dictionary, DML Precompiler, and DDL Compiler. [cite: 3999, 4000]

5.4 Types of Relations

    [cite_start]
  • Base Tables: A table that physically exists in the database. [cite: 4000]
    • [cite_start]
  • Query Results: The (virtual) table of results from a query. [cite: 4000]
  • Views: A virtual table based on a query, but stored in the database as an object. [cite_start]Views provide security and query simplicity. [cite: 4000, 4001]

5.5 Properties of Relations

    [cite_start]
  • Atomic Values in Fields: No multi-valued attributes. [cite: 4001]
    • [cite_start]
  • Entries from Same Domain: All entries in a column are of the same type. [cite: 4001]
    • [cite_start]
  • Unique Tuples: Each row is unique, guaranteed by a primary key. [cite: 4001]
    • [cite_start]
  • Unique Attribute Name: Each column has a unique name. [cite: 4001]
    • [cite_start]
  • Insignificant Attribute Sequence: The order of columns doesn't matter. [cite: 4001]
    • [cite_start]
  • Insignificant Tuple Sequence: The order of rows doesn't matter. [cite: 4001]

5.6 Codd's Rules

[cite_start]

A set of 12 rules (0-12) proposed by E.F. Codd to define what makes a DBMS truly "relational." [cite: 4002] [cite_start]Key rules include the Information Rule (all data in tables), Guaranteed Access (all data accessible by table/PK/attribute), and Systematic Treatment of Nulls. [cite: 4002]

5.7 Relational Data Integrity

    [cite_start]
  • Entity Integrity: Ensures the primary key can never be null. [cite: 4003]
    • [cite_start]
  • Domain Integrity: Enforces that all values in a column are from its defined domain (e.g., "Grades" must be A, B, C, D, or F). [cite: 4003]
  • Referential Integrity: Ensures that a foreign key value must either be null or must match a valid, existing primary key value in the parent table. [cite_start]This prevents "orphan" records. [cite: 4004]

5.8 Database Languages

    [cite_start]
  • Data Definition Language (DDL): Used to define the database schema (e.g., CREATE TABLE, ALTER TABLE). [cite: 4004]
    • [cite_start]
  • Data Manipulation Language (DML): Used to manipulate data within the schema (e.g., INSERT, UPDATE, DELETE, SELECT). [cite: 4005]

5.9 Relational Algebra

[cite_start]

A procedural query language that defines operations on relations (tables) to produce new relations. [cite: 4005]

[cite_start]

Unary Operations (one relation): [cite: 4005]

  • Selection (): Selects rows (tuples) that satisfy a condition. (Filters horizontally) [cite_start][cite: 4006]
  • Projection (): Selects columns (attributes). (Filters vertically) [cite_start][cite: 4007]
[cite_start]

Binary Operations (two relations): [cite: 4005]

    [cite_start]
  • Union (): Combines all rows from two compatible tables. [cite: 4008]
    • [cite_start]
  • Difference (-): Returns rows that are in the first table but not the second. [cite: 4009]
    • [cite_start]
  • Intersection (): Returns rows that appear in both tables. [cite: 4009]
    • [cite_start]
  • Product (x): Creates a new table of all possible combinations of rows from two tables. [cite: 4009]
    • [cite_start]
  • Division (): Finds rows in one table that match all rows in another. [cite: 4010]

5.10 Joins

[cite_start]

A join operation combines data from two or more tables into a single result table based on matching criteria (the join condition). [cite: 4010]

  • Theta Join: A join that links tables based on any comparison operator (theta), such as =, <, >, or !=. [cite_start]It is a general-purpose join. [cite: 4010]
  • Equi Join: A type of Theta Join that only uses the equality operator (=) as its join condition. This is the most common type of join. [cite_start]The common attributes will appear twice in the result table. [cite: 4011]
  • Natural Join: A special type of Equi Join that joins tables on *all* attributes that have the same name. [cite_start]The duplicate columns are removed from the final result. [cite: 4011]
    • [cite_start]
  • Outer Join: A join that retrieves all rows that match the join condition, *plus* all rows from one or both tables that do not have a match. [cite: 4012]
      [cite_start]
    • Left Outer Join: Includes all rows from the *left* table and matching rows from the right table (or NULLs if no match). [cite: 4012]
      • [cite_start]
    • Right Outer Join: Includes all rows from the *right* table and matching rows from the left table (or NULLs if no match). [cite: 4012]
      • [cite_start]
    • Full Outer Join: Includes all rows from *both* tables, filling with NULLs where matches are not found. [cite: 4012]
  • Semi Join: A join that returns only the rows from the first table for which a match is found in the second table. [cite_start]It does not include any columns from the second table. [cite: 4013]

5.14 Normalization

[cite_start]

The process of producing a simpler, more reliable database structure by breaking down complex tables to reduce redundancy and eliminate the anomalies (insertion, deletion, modification). [cite: 4015, 4033] [cite_start]This relies on the concept of Functional Dependency. [cite: 4016]

5.15 Functional Dependency

A relationship between attributes. [cite_start]An attribute B is functionally dependent on A if the value of A determines the value of B (written A -> B). [cite: 4016] [cite_start]The attribute on the left side (A) is called the determinant. [cite: 4016]

Normalization Forms

    [cite_start]
  • 5.16 First Normal Form (1NF): A relation is in 1NF if it does not contain a repeating group. [cite: 4017] [cite_start]All attributes must be atomic (each cell has only one value). [cite: 4017]
    • [cite_start]
  • 5.17 Full Functional Dependency: An attribute B is fully functionally dependent on A if it is dependent on A, but not on any proper subset of A. [cite: 4018]
    • [cite_start]
  • 5.18 Second Normal Form (2NF): Must be in 1NF, AND all non-key attributes must be fully functionally dependent on the entire primary key (no partial dependencies). [cite: 4018] (This is only an issue for tables with composite primary keys).
  • 5.19 Transitive Dependency: A condition where an attribute A determines B, and B determines C, but B is *not* part of the primary key. [cite_start]This means a non-key attribute (C) is dependent on another non-key attribute (B). [cite: 4020]
    • [cite_start]
  • 5.20 Third Normal Form (3NF): Must be in 2NF, AND no non-key attribute is transitively dependent on the primary key (no transitive dependencies). [cite: 4020]
  • 5.21 Boyce-Codd (BCNF): A stricter 3NF. [cite_start]A relation is in BCNF if and only if every determinant is a candidate key. [cite: 4022]
    • [cite_start]
  • 5.22 Fourth Normal Form (4NF): Must be in BCNF, AND have no multi-valued dependencies (where one attribute determines a set of multiple values for another attribute, e.g., one StudentID -> {Math, Reading} and StudentID -> {Tennis, Jogging}). [cite: 4022, 4023]

Normalization Projects

Normalization Project 1 (pg 135)

[cite_start]

A table `Student_in_Society` with `StdID`, `StdName`, and a repeating group for `SocietyID`, `SocName`, `SupID`, `Supervisor`, `Position`. [cite: 4026]

Process:

    [cite_start]
  • 1NF: Remove repeating groups by creating two tables: `Student(StdID, StdName)` and `Student_in_Society(StdID, SocietyID, SocName, SupID, Supervisor, Position)`. [cite: 4026]
  • 2NF: The 2nd table has partial dependencies. `SocName`, `SupID`, and `Supervisor` depend only on `SocietyID`. `Position` depends only on `StdID` and `SocietyID`. This is broken into:
      [cite_start]
    • `Society(SocietyID, SocName, SupID, Supervisor)` [cite: 4027]
      • [cite_start]
    • `Student_in_Society(StdID, SocietyID, Position)` [cite: 4027]
  • 3NF: The new `Society` table has a transitive dependency: `SocietyID -> SupID -> Supervisor`. This is broken into:
      [cite_start]
    • `Society(SocietyID, SocName, SupID)` [cite: 4027]
      • [cite_start]
    • `Society_Teacher(SupID, Supervisor)` [cite: 4028]

Normalization Project 2 (pg 137)

[cite_start]

A table with `StudentID`, `StudentName`, `CourseID`, `CourseLength`, and a repeating group for `UnitCode`, `UnitName`, `Lecturer`. [cite: 4028]

Process:

    [cite_start]
  • 1NF: Remove repeating groups by creating two tables: `Student(StudentID, StudentName, CourseID, CourseLength)` and `Unit(StudentID, UnitCode, UnitName, Lecturer)`. [cite: 4029]
  • 2NF: The new `Unit` table has partial dependencies. `UnitName` and `Lecturer` depend only on `UnitCode`. This is broken into:
      [cite_start]
    • `UnitTable(UnitCode, UnitName, Lecturer)` [cite: 4029]
      • [cite_start]
    • `Take(StudentID, UnitCode)` [cite: 4029]
  • 3NF: The `Student` table has a transitive dependency: `StudentID -> CourseID -> CourseLength`. This is broken into:
      [cite_start]
    • `StudentTable(StudentID, StudentName, CourseID)` [cite: 4030]
      • [cite_start]
    • `CourseTable(CourseID, CourseLength)` [cite: 4030]
