Chapter 1: Introduction to Databases

1.1 Data

A collection of raw, unprocessed facts and figures. Data is collected from different sources for different purposes and is not yet processed to get its exact meaning.

1.1.2 Types of Data:

  • Numeric: Digits 0-9 (e.g., 10, 245).
  • Alphabetic: Letters A-Z and blank spaces (e.g., "IT Series").
  • Alphanumeric: Digits, letters, and special characters (e.g., "H#17", "$300").
  • Image: Charts, graphs, and pictures.
  • Audio: Music, speech, and any sound.
  • Video: Full-motion images used to display actions.

1.2 Information

Processed data. It is an organized and processed form of data, making it meaningful and useful for decision-making. Data is the input, and information is the output.

1.3 Difference between Data and Information

Data Information
Consists of unprocessed raw facts. Is the processed form of data.
Used as input to the computer. Is the output of the computer.
Is not meaningful. Is meaningful.
Not used in decision-making. Very important for decision-making.

1.4 Metadata

Data about data. It describes the properties and characteristics of other data, such as its size, format, rules, and constraints (e.g., data type, length, "must contain @").

1.5 File Processing System

The traditional computer-based method where each department in an organization has its own set of files, designed specifically for their own applications. This system has many disadvantages.

1.5.1 Disadvantages of File Processing System:

  • Data Redundancy and Inconsistency: The same data is duplicated in multiple files, which can lead to conflicts (e.g., an address is updated in one file but not another).
  • Data Isolation: Data is stored in various files, making it very difficult to write new applications to retrieve data from multiple files.
  • Integrity Problems: It's hard to enforce consistency constraints (e.g., "Marks must be a number").
  • Program-Data Dependence: Programs are tied to the file format. If you change the file's structure (e.g., add a field), you must change all programs that use that file.
  • Atomicity Problem: A transaction (e.g., a money transfer) must happen entirely or not at all. File systems can fail mid-step, leaving data inconsistent.
  • Security Problems: It's difficult to provide different levels of access (e.g., read-only vs. write) for different users.
  • Program Maintenance: Programs are difficult to maintain.

1.6 Database

An organized collection of related data stored in an efficient and compact manner. Data is typically arranged in Tables (relations), which consist of Rows (records or tuples) and Columns (fields or attributes).

Examples: A phone directory, a library card catalog, or an accounting system.

1.7 Database Management System (DBMS)

A collection of programs (software) used to create and maintain a database. The DBMS is the general-purpose tool that provides the facilities to define, store, and manipulate the data.

1.8 Components of Database Environment

  • Repository: A collection of all data definitions, data relationships, and report formats (i.e., the metadata).
  • DBMS: The software that manages the database.
  • Database: The organized collection of related data.
  • Application Program: A program used to send commands to the DBMS to manipulate the database.
  • User Interface: The visual environment (forms, menus, reports) that users interact with.
  • Data Administrators: Persons responsible for the whole information system.
  • System Analysts & Application Programmers: Determine user requirements and implement them as programs.
  • End Users: The people who interact with the application to insert, delete, and update data.

1.9 Database Approach

1.9.1 Advantages

  • Redundancy Control
  • Data Consistency
  • Consistency Constraints
  • Data Atomicity
  • Data Security
  • Reduced Development Time
  • Compactness
  • Easier Reporting
  • Data Sharing
  • Increased Concurrency
  • Improved Backup and Recovery
  • Data Independence

1.9.2 Disadvantages

  • High Cost of DBMS
  • Higher Hardware Cost
  • Higher Programming Cost
  • High Conversion Cost
  • More Chance of Failure (centralized)
  • Complexity & Performance

1.10 Difference between File and Database Approach

The key difference is Data Independence. In a file-based approach, programs and data are inter-dependent. In a database approach, they are independent. This means the database structure can be changed without needing to rewrite all the application programs.

1.11 Application Program

A program that sends commands to the DBMS to manipulate data. The user interacts with the application program (the "front-end"), and the application program interacts with the DBMS (which manages the "back-end" database).

[Image of relationship: User -> User Interface -> Application Program -> DBMS -> Database]

1.12 Range of Database Applications

  • Personal Computer Databases: For a single user (e.g., on a laptop).
  • Workgroup Databases: For a small team (less than 25) connected via a Local Area Network (LAN).
  • Department Databases: For a functional unit (25-100 people), like marketing or accounting.
  • Enterprise Databases: Supports the entire organization, often by integrating data from other databases (e.g., a Data Warehouse).

1.13 Types of Users

  • Application Programmers: Professionals who write the programs that interact with the database.
  • End Users: People who use the applications directly.
    • Naive Users: Have no technical knowledge and use the system via simple menus and buttons.
    • Sophisticated Users: Are familiar with the database structure and can use a query language (like SQL) directly.
  • Database Administrator (DBA): The most technical user, responsible for managing the entire database system (design, security, backups, performance).

Chapter 2: The Database Environment

2.1 Three-Level Architecture

The fundamental architecture for modern database systems, it separates the database into three levels or views to provide data independence.

  • 1. External Level / View: What a particular user or application sees. It's a "subschema" showing only the part of the database relevant to that user. There can be many external views.
  • 2. Logical or Conceptual Level / View: The "community view." A complete logical description of the *entire* database (all entities, attributes, relationships). It hides physical storage details. There is only one conceptual schema.
  • 3. Internal or Physical Level: Describes *how* the data is physically stored on the storage media (data structures, file organization, indexes). There is only one internal schema.

2.2 Mapping

The process of converting or relating data between the three levels.

  • External/Conceptual Mapping: Defines the correspondence between a specific external view and the conceptual view.
  • Conceptual/Internal Mapping: Defines the correspondence between the conceptual view and the stored database.

2.3 Data Independence

The main advantage of the 3-Level Architecture. It means you can change the schema at one level without having to change the schema at the next level up.

  • Physical Data Independence: The ability to change the Internal level (e.g., add an index, change file organization) without affecting the Conceptual level. This is relatively easy to achieve.
  • Logical Data Independence: The ability to change the Conceptual level (e.g., add a new entity or attribute) without affecting existing External views. This is much harder to achieve.

2.4 Data Models

A collection of concepts used to describe and manipulate data, relationships, and constraints. A data model has three parts:

  • Structural Part: Rules for how a database can be developed.
  • Manipulative Part: Defines the operations (e.g., update, retrieve) that can be performed.
  • Set of Integrity Rules: Ensures the accuracy of data.

2.5 Types of Data Models

2.5.2 Record-Based Data Models

Models based on records (tables).

  • 2.5.2.2 Network Model: An extension of the hierarchical model. A child (member) can have multiple parents (owners) through "sets". More flexible but very complex.
  • 2.5.2.3 Relational Model: Represents data in 2D tables (relations). This is the standard for almost all modern DBMSs.

Other Models

  • 2.5.1 Object-Based Data Models: Use concepts like entities, attributes, and relationships (e.g., the E-R Model, Semantic Model, Object-Oriented Model).
  • 2.5.3 Physical Data Models: Describe data at the internal level (how data is stored), e.g., unifying model, frame memory.

2.6 Functions of DBMS

  • Data Processing: Creating, storing, and arranging data.
  • User-accessible Catalog: Maintains a data dictionary of metadata.
  • Transaction Support: Ensures transactions are atomic (all or nothing).
  • Concurrency Support: Allows multiple users to access data simultaneously without conflicts.
  • Recovery Services: Can recover the database to a consistent state after a failure.
  • Data Communication Support: Integrates with network software.
  • Integrity Services: Enforces data constraints to maintain accuracy.
  • Authorization Services: Controls who can access what data.

2.7 Database Development Process

Strategies for developing a database application:

  • Top-Down Development: Starts with general goals and moves to specific issues.
  • Bottom-Up Development: Starts with specific issues (a single system) and moves to general ones.

2.8 System Development Life Cycle (SDLC)

The conventional way to develop an information system.

[Image of SDLC waterfall diagram: Preliminary Investigation -> Analysis -> Design -> Development -> Testing -> Implementation -> Maintenance]
  • 2.8.1 Preliminary Investigation: Identify problems and assess feasibility (technical, economic).
  • 2.8.2 Requirement Analysis: Study the current business system in detail to find out how it works.
  • 2.8.3 System Design: Define *how* the new system will meet the requirements (forms, reports, database design).
  • 2.8.4 Software Development: The actual coding of the programs.
  • 2.8.5 System Testing: Integrating and testing all modules.
  • 2.8.6 System Implementation: Installing the system and training users.
  • 2.8.7 System Maintenance: Modifying and enhancing the system after it's in use.

2.9 Staged Database Design Approach

A top-down approach to database design.

  • 1. Analyze User Environment
  • 2. Develop Logical Data Model
  • 3. Choose a DBMS
  • 4. Map Logical Model to DBMS
  • 5. Develop Physical Model
  • 6. Evaluate Physical Model
  • 7. Perform Tuning
  • 8. Implement Physical Model

Chapter 3: The Entity-Relationship (E-R) Model

3.1 & 3.2 Elements of the E-R Model

The E-R model (by Peter Chen, 1976) is a logical, visual representation of data. It's simple, provides a clear visual representation, and is an effective communication tool.

  • 3.2.1 Entity: A person, place, thing, or event about which data is collected (e.g., STUDENT, BOOK).
    • Entity Type: A set of entities with the same attributes (e.g., the STUDENT class).
    • Entity Instance: A single member of the set (e.g., the student 'Abdullah').
    • Entity Set: All entities of a particular type at a given time.
  • 3.2.2 Attribute: A characteristic or property of an entity (e.g., Name, Address).
    • Attribute Domain: The set of possible values for an attribute (e.g., GPA is 0-4).
  • 3.2.3 Relationship: A logical connection or association between entities (e.g., a STUDENT "Studies" a COURSE).

3.3 E-R Diagram (Types of Attributes & Entities)

3.3.1 Types of Attributes

Simple (Atomic): Cannot be subdivided (e.g., Gender).
Composite: Can be subdivided (e.g., Address -> Street, City).
Single-Valued: Has only one value (e.g., Age).
Multi-Valued: Can have many values (e.g., Skills). Represented by double-line oval.
Stored: Physically stored in the database (e.g., DateOfBirth).
Derived: Calculated from a stored attribute (e.g., Age is derived from DateOfBirth). Represented by dashed-line oval.
3.3.1.7 Identifiers (Key): An attribute that uniquely identifies one instance (e.g., StudentID).
  • Unique vs. Non-unique: EmployeeID is unique. Department is non-unique (identifies a set).
  • Simple vs. Composite: Student_ID is a simple identifier. OrderID (OrderNo + Date) is composite.

3.3.2 Types of Entities

  • Weak Entities & Strong Entities: A Strong Entity (or parent/owner) can exist on its own and has its own unique identifier (e.g., PERSON). A Weak Entity (or child/dependent) cannot exist without its owner (e.g., CHILD depends on PERSON). Represented by a double-line rectangle and a double-line identifying relationship.
  • Associative Entity: An entity that associates the instances of one or more entity types. Often used to "resolve" a many-to-many relationship and store attributes about that relationship (e.g., an AWARD links a SCHOLAR and RESEARCH).

3.4 Degree of Relationships

  • 3.4.1 Unary (Recursive): A relationship between instances of the same entity type (e.g., a PERSON "Lives With" another PERSON).
  • 3.4.2 Binary: A relationship between two entity types. This is the most common.
    • One-to-One (1:1): A CHAIRMAN "Runs" one DEPARTMENT.
    • One-to-Many (1:N): A COLLEGE "Has" many DEPARTMENTS.
    • Many-to-Many (M:N): A STUDENT "Studies" many COURSES.
  • 3.4.3 Ternary: A relationship between three entity types (e.g., a CONSULTANT, CLIENT, and CONTRACT are all linked by a "Signs" relationship).

3.4.4 Cardinality Constraints

Specifies the number of instances of one entity that can be associated with an instance of another entity. It defines the "business rules."

  • Maximum Cardinality: The maximum number (1 or Many).
  • Minimum Cardinality: The minimum number (0 or 1).
    • If minimum is 0, participation is Optional.
    • If minimum is 1, participation is Mandatory.

3.5 Subtype & Supertype Entities

A way to model IS-A relationships (e.g., "A STUDENT IS-A PERSON"). A supertype contains common attributes, while subtypes contain specific attributes and inherit from the supertype.

  • Supertype: The general entity (e.g., PERSON) that contains common attributes (Name, Address).
  • Subtype: The specific entity (e.g., STUDENT) that contains specific attributes (RollNo) and inherits all attributes from the supertype.
  • 3.5.1 Generalization: The process of identifying a more general entity type (bottom-up).
  • 3.5.2 Specialization: The process of identifying more specific entity types (top-down).

Chapter 4: The Semantic Object Model

4.1 & 4.2 Semantic Object Model

An alternative to the E-R model that views the system as a collection of objects. It's considered more closely aligned with a user's perception of the world.

A Semantic Object is a "thing" in the user's environment, grouped into classes (e.g., STUDENT class). It has attributes:

  • 4.2.1.1 Simple Attributes: Contain a single value (e.g., Roll No).
  • 4.2.1.2 Group Attributes: Composites of other attributes (e.g., Address).
  • 4.2.1.3 Semantic Object Attributes: An attribute that *is* another semantic object (e.g., a COLLEGE object has a PROFESSOR attribute, which is a link to the PROFESSOR object).

4.2.2 Attribute Cardinality

Specified in n.m format, where 'n' is the minimum (0 or 1) and 'm' is the maximum (1 or N). E.g., 1.1 (mandatory one), 0.N (optional many).

4.2.4 Object Identifiers

An attribute or collection of attributes used to identify an object instance (e.g., Roll No). Can be unique (like a Roll No) or non-unique (like a Name).

4.3 Attribute Domains

A collection of all possible values for an attribute. A domain has two descriptions:

  • Physical Description: The data type (e.g., "string", "Roll No should not exceed 99").
  • Semantic Description: The function or purpose of the attribute (e.g., "Name of a student must be a valid name").

4.4 Semantic Object Views

A part of an object that is displayed in a particular application. One object can be displayed in many different views (e.g., one view for students, one for professors, both from the same COLLEGE object).

4.5 Types of Objects

  • 4.5.1 Simple Object: Contains only single-valued, non-object attributes.
  • 4.5.2 Composite Object: Contains one or more multi-valued, non-object attributes.
  • 4.5.3 Compound Object: Contains at least one object attribute.
  • 4.5.4 Hybrid Object: A combination of composite and compound.
  • 4.5.5 Association Object: An object that links two or more other objects, typically storing data about that relationship.
  • 4.5.6 Parent/Subtype Objects: A subtype object inherits the attributes of its parent (similar to E-R super/subtypes).
  • 4.5.7 Archetype/Version Objects: An archetype object produces other objects that are "versions" of it (e.g., BOOK is the archetype, EDITION is the version).

Chapter 5: Relational Model & Normalization

5.1 Relational Model

The standard data model (by E.F. Codd, 1970) where data is stored in tables. Advantages include Data Integrity, Data Independence, and Structural Independence.

5.1.1 Terminology:

  • Relation: A table.
  • Tuple: A row (a single record).
  • Attribute: A column (a field).
  • Domain: The set of all possible values for an attribute.
  • Degree: The number of attributes (columns).
  • Cardinality: The number of tuples (rows).

5.2 Keys

Keys are attributes used to identify tuples (rows) in a relation.

  • 5.2.1 Super Key: Any attribute or set of attributes that uniquely identifies a tuple.
  • 5.2.2 Candidate Key: A minimal super key (no attribute can be removed and it still be unique).
  • 5.2.3 Primary Key (PK): The one candidate key chosen by the designer to be the main identifier. It cannot contain null values.
  • 5.2.4 Alternate Key: A candidate key that was *not* chosen to be the primary key.
  • 5.2.5 Composite Key: A primary key that consists of two or more attributes.
  • 5.2.6 Foreign Key (FK): An attribute in one relation whose values match the primary key in another relation, used to link the tables.

5.3 Relational Database Management System (RDBMS)

A RDBMS is the software that manages relational databases. Its components include a File Manager, Database Manager, Query Processor, Data Dictionary, DML Precompiler, and DDL Compiler.

5.4 Types of Relations

  • 5.4.1 Base Tables: A table that physically exists in the database.
  • 5.4.2 Query Results: The (virtual) table of results from a query.
  • 5.4.3 Views: A virtual table based on a query, but stored in the database as an object. Views provide security and query simplicity.

5.5 Properties of Relations

  • 1. Atomic Values in Fields: No multi-valued attributes.
  • 2. Entries from Same Domain: All entries in a column are of the same type.
  • 3. Unique Tuples: Each row is unique, guaranteed by a primary key.
  • 4. Unique Attribute Name: Each column has a unique name.
  • 5. Insignificant Attribute Sequence: The order of columns doesn't matter.
  • 6. Insignificant Tuple Sequence: The order of rows doesn't matter.

5.6 Codd's Rules

A set of 12 rules (0-12) proposed by E.F. Codd to define what makes a DBMS truly "relational." Key rules include the Information Rule (all data in tables), Guaranteed Access (all data accessible by table/PK/attribute), Systematic Treatment of Nulls, and a Comprehensive Data Sub-language (like SQL).

5.7 Relational Data Integrity

  • 5.7.1 Entity Integrity: Ensures the primary key can never be null.
  • 5.7.2 Domain Integrity: Enforces that all values in a column are from its defined domain (e.g., "Grades" must be A, B, C, D, or F).
  • 5.7.3 Referential Integrity: Ensures that a foreign key value must either be null or must match a valid, existing primary key value in the parent table. This prevents "orphan" records.

5.8 Database Languages

  • 5.8.1 Data Definition Language (DDL): Used to define the database schema (e.g., CREATE TABLE, ALTER TABLE).
  • 5.8.2 Data Manipulation Language (DML): Used to manipulate data within the schema (e.g., INSERT, UPDATE, DELETE, SELECT).

5.9 Relational Algebra

A procedural query language that defines operations on relations (tables) to produce new relations.

5.9.1 Basic Operations:

  • Unary Operations (one relation):
    • 5.9.2 Selection (): Selects rows (tuples) that satisfy a condition. (Filters horizontally)
    • 5.9.3 Projection (): Selects columns (attributes). (Filters vertically)
  • Binary Operations (two relations):
    • 5.9.4 Set Operations:
      • Union (): Combines all rows from two compatible tables.
      • Difference (-): Returns rows that are in the first table but not the second.
      • Intersection (): Returns rows that appear in both tables.
      • Product (x): Creates a new table of all possible combinations of rows from two tables.
      • Division (): Finds rows in one table that match all rows in another.

5.10 Joins

A join operation combines data from two or more tables into a single result table based on matching criteria (the join condition).

  • 5.10.1 Theta Join: A join that links tables based on any comparison operator (theta), such as =, <, >, or !=. It is a general-purpose join.
  • 5.10.2 Equi Join: A type of Theta Join that only uses the equality operator (=) as its join condition. This is the most common type of join. The common attributes will appear twice in the result table.
  • 5.10.3 Natural Join: A special type of Equi Join that joins tables on *all* attributes that have the same name. The duplicate columns are removed from the final result.
  • 5.10.4 Outer Join: A join that retrieves all rows that match the join condition, *plus* all rows from one or both tables that do not have a match.
    • Left Outer Join: Includes all rows from the *left* table and matching rows from the right table (or NULLs if no match).
    • Right Outer Join: Includes all rows from the *right* table and matching rows from the left table (or NULLs if no match).
    • Full Outer Join: Includes all rows from *both* tables, filling with NULLs where matches are not found.
  • 5.10.5 Semi Join: A join that returns only the rows from the first table for which a match is found in the second table. It does not include any columns from the second table.

5.12 Relational Algebra vs. Relational Calculus

Relational Algebra (procedural) and Relational Calculus (non-procedural) have the same expressive power. Relational Algebra is considered higher-level than languages like SQL because it specifies the order of operations, whereas SQL allows the compiler to determine the most efficient order.

5.14 Normalization

The process of producing a simpler, more reliable database structure by breaking down complex tables to reduce redundancy and eliminate anomalies. The goal is to create a set of relations free of processing problems.

5.15 Functional Dependency

A relationship between attributes. An attribute B is functionally dependent on A if the value of A determines the value of B (written A -> B). The attribute on the left side (A) is called the determinant.

Normalization Forms

  • 5.16 First Normal Form (1NF): A relation is in 1NF if it does not contain a repeating group. All attributes must be atomic (each cell has only one value).
  • 5.17 Full Functional Dependency: An attribute B is fully functionally dependent on A if it is dependent on A, but not on any proper subset of A.
  • 5.18 Second Normal Form (2NF): Must be in 1NF, AND all non-key attributes must be fully functionally dependent on the entire primary key (no partial dependencies). (This is only an issue for tables with composite primary keys).
  • 5.19 Transitive Dependency: A condition where an attribute A determines B, and B determines C, but B is *not* part of the primary key. This means a non-key attribute (C) is dependent on another non-key attribute (B).
  • 5.20 Third Normal Form (3NF): Must be in 2NF, AND no non-key attribute is transitively dependent on the primary key (no transitive dependencies).
  • 5.21 Boyce-Codd (BCNF): A stricter 3NF. A relation is in BCNF if and only if every determinant is a candidate key.
  • 5.22 Fourth Normal Form (4NF): Must be in BCNF, AND have no multi-valued dependencies (where one attribute determines a set of multiple values for another attribute).

Normalization Project 1 (pg 135)

A table Student_in_Society with StdID, StdName, and a repeating group for SocietyID, SocName, SupID, Supervisor, Position.

Process:

  • 1NF: Remove repeating groups by creating two tables: Student(StdID, StdName) and Student_in_Society(StdID, SocietyID, SocName, SupID, Supervisor, Position).
  • 2NF: The 2nd table has partial dependencies. SocName, SupID, and Supervisor depend only on SocietyID. This is broken into:
    • Society(SocietyID, SocName, SupID, Supervisor)
    • Student_in_Society(StdID, SocietyID, Position)
  • 3NF: The new Society table has a transitive dependency: SocietyID -> SupID -> Supervisor. This is broken into:
    • Society(SocietyID, SocName, SupID)
    • Society_Teacher(SupID, Supervisor)
