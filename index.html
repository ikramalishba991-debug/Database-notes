Chapter 1: Introduction to Databases

1.1 Data

A collection of raw, unprocessed facts and figures. Data is collected from different sources for different purposes and is not yet processed to get its exact meaning.

Types of Data:

  • Numeric: Digits 0-9 (e.g., 10, 245).
  • Alphabetic: Letters A-Z and blank spaces (e.g., "IT Series").
  • Alphanumeric: Digits, letters, and special characters (e.g., "H#17").
  • Image: Charts, graphs, and pictures.
  • Audio: Music, speech, and any sound.
  • Video: Full-motion images.

1.2 Information

Processed data. It is an organized and processed form of data, making it meaningful and useful for decision-making. Data is the input, and information is the output.

1.3 Difference between Data and Information

Data Information
Consists of unprocessed raw facts. Is the processed form of data.
Used as input to the computer. Is the output of the computer.
Is not meaningful. Is meaningful.
Not used in decision-making. Very important for decision-making.

1.4 Metadata

Data about data. It describes the properties and characteristics of other data, such as its size, format, rules, and constraints (e.g., data type, length, "must contain @").

1.5 File Processing System

The traditional computer-based method where each department in an organization has its own set of files, designed specifically for their own applications. This system has many disadvantages.

Disadvantages:

  • Data Redundancy and Inconsistency: The same data is duplicated in multiple files, which can lead to conflicts (e.g., an address is updated in one file but not another).
  • Data Isolation: Data is stored in various files, making it very difficult to write new applications to retrieve data from multiple files.
  • Integrity Problems: It's hard to enforce consistency constraints (e.g., "Marks must be a number").
  • Program-Data Dependence: Programs are tied to the file format. If you change the file's structure (e.g., add a field), you must change all programs that use that file.
  • Atomicity Problem: A transaction (e.g., a money transfer) must happen entirely or not at all. File systems can fail mid-step, leaving data inconsistent.
  • Security Problems: It's difficult to provide different levels of access (e.g., read-only vs. write) for different users.
  • Program Maintenance: Most of the budget is spent on maintaining old programs rather than building new ones.

1.6 Database

An organized collection of related data stored in an efficient and compact manner. Data is typically arranged in Tables (relations), which consist of Rows (records or tuples) and Columns (fields or attributes).

Examples: A phone directory, a library card catalog, or an accounting system.

1.7 Database Management System (DBMS)

A collection of programs (software) used to create and maintain a database. The DBMS is the general-purpose tool that provides the facilities to define, store, and manipulate the data.

1.8 Components of Database Environment

  • Repository: A collection of all data definitions, relationships, and formats (i.e., the metadata).
  • DBMS: The software that manages the database.
  • Database: The organized collection of data.
  • Application Program: A program used to send commands to the DBMS (e.g., Developer2000, Power Builder).
  • User Interface: The visual environment (forms, menus, reports) that users interact with.
  • Data Administrators (DA): Persons responsible for the whole information system.
  • System Analysts & Programmers: Those who design and implement the applications.
  • End Users: The people who interact with the application to insert, delete, and update data.

1.9 Database Approach

Advantages

  • Redundancy Control
  • Data Consistency
  • Consistency Constraints
  • Data Atomicity
  • Data Security
  • Reduced Development Time
  • Compactness
  • Easier Reporting
  • Data Sharing
  • Increased Concurrency
  • Improved Backup and Recovery
  • Data Independence

Disadvantages

  • High Cost of DBMS
  • Higher Hardware Cost
  • Higher Programming Cost
  • High Conversion Cost
  • More Chance of Failure (centralized)
  • Complexity & Performance (can be slower)

1.10 Difference between File and Database Approach

The key difference is Data Independence. In a file-based approach, programs and data are inter-dependent. In a database approach, they are independent. This means the database structure can be changed without needing to rewrite all the application programs.

1.11 Application Program

A program that sends commands to the DBMS to manipulate data. The user interacts with the application program (the "front-end"), and the application program interacts with the DBMS (which manages the "back-end" database).

1.12 Range of Database Applications

  • Personal Computer Databases: For a single user (e.g., on a laptop).
  • Workgroup Databases: For a small team (less than 25) connected via a Local Area Network (LAN).
  • Department Databases: For a functional unit (25-100 people), like marketing or accounting.
  • Enterprise Databases: Supports the entire organization, often by integrating data from other databases (e.g., a Data Warehouse).

1.13 Types of Users

  • Application Programmers: Professionals who write the programs that interact with the database.
  • End Users: People who use the applications directly.
    • Naive Users: Have no technical knowledge and use the system via simple menus and buttons.
    • Sophisticated Users: Are familiar with the database structure and can use a query language (like SQL) directly.
  • Database Administrator (DBA): The most technical user, responsible for managing the entire database system (design, security, backups, performance).

1.14 History of Database Systems

  • 1960s: Computers became cost-effective. Two main models developed: Network model (CODASYL) and Hierarchical (IMS).
  • 1970-72: E.F. Codd proposed the Relational Model, a landmark paper that became the standard.
  • 1976: P. Chen proposed the Entity-Relationship (E-R) model for database design.
  • Mid-1980s: SQL (Structured Query Language) became the standard.
  • Early 1990s: Rise of client-server tools (PowerBuilder, Oracle Developer).
  • Late 1990s: The Internet boom led to Web/DB connectors (JDBC, ASP, etc.).
  • Present: Focus on huge (terabyte) systems, data mining, data warehousing, mobile databases, and XML.

Chapter 2: The Database Environment

2.1 Three-Level Architecture

The fundamental architecture for modern database systems, it separates the database into three levels or views to provide data independence.

  • 1. External Level / View: What a particular user or application sees. It's a "subschema" showing only the part of the database relevant to that user. There can be many external views.
  • 2. Logical or Conceptual Level / View: The "community view." A complete logical description of the *entire* database (all entities, attributes, relationships). It hides physical storage details. There is only one conceptual schema.
  • 3. Internal or Physical Level: Describes *how* the data is physically stored on the storage media (data structures, file organization, indexes). There is only one internal schema.

2.2 Mapping

The process of converting or relating data between the three levels.

  • External/Conceptual Mapping: Defines the correspondence between a specific external view and the conceptual view.
  • Conceptual/Internal Mapping: Defines the correspondence between the conceptual view and the stored database.

2.3 Data Independence

The main advantage of the 3-Level Architecture. It means you can change the schema at one level without having to change the schema at the next level up.

  • Physical Data Independence: The ability to change the Internal level (e.g., add an index, change file organization) without affecting the Conceptual level. This is relatively easy to achieve.
  • Logical Data Independence: The ability to change the Conceptual level (e.g., add a new entity or attribute) without affecting existing External views. This is much harder to achieve.

2.4 Data Models

A collection of concepts used to describe and manipulate data, relationships, and constraints. A data model has three parts:

  • Structural Part: Rules for how a database can be developed.
  • Manipulative Part: Defines the operations (e.g., update, retrieve) that can be performed.
  • Set of Integrity Rules: Ensures the accuracy of data.

2.5 Types of Data Models

Record-Based Data Models

Models based on records (tables).

  • Hierarchical Model: Organizes data in a parent-child tree structure. A child can only have one parent. Very fast but inflexible.
    [Image of Hierarchical Model diagram]
  • Network Model: An extension of the hierarchical model. A child (member) can have multiple parents (owners). More flexible but very complex.
  • Relational Model: Represents data in 2D tables (relations). This is the standard for almost all modern DBMSs.

Other Models

  • Object-Based Data Models: Use concepts like entities, attributes, and relationships (e.g., the E-R Model) or objects (e.g., Object-Oriented Model).
  • Physical Data Models: Describe data at the internal level (how data is stored).

2.6 Functions of DBMS

  • Data Processing: Creating, storing, and arranging data.
  • User-accessible Catalog: Maintains a data dictionary of metadata.
  • Transaction Support: Ensures transactions are atomic (all or nothing).
  • Concurrency Support: Allows multiple users to access data simultaneously without conflicts.
  • Recovery Services: Can recover the database to a consistent state after a failure.
  • Data Communication Support: Integrates with network software.
  • Integrity Services: Enforces data constraints to maintain accuracy.
  • Authorization Services: Controls who can access what data.

2.7 & 2.8 Database Development Process (SDLC)

The System Development Life Cycle (SDLC) is the conventional way to develop an information system.

[Image of SDLC waterfall diagram: Preliminary Investigation -> Analysis -> Design -> Development -> Testing -> Implementation -> Maintenance]
  • Preliminary Investigation: Identify problems and assess feasibility.
  • Requirement Analysis: Study the current system and user needs in detail.
  • System Design: Define *how* the new system will meet the requirements (forms, reports, database design).
  • Software Development: The actual coding of the programs.
  • System Testing: Integrating and testing all modules.
  • System Implementation: Installing the system and training users.
  • System Maintenance: Modifying and enhancing the system after it's in use.

2.10 Design Tools: Data Flow Diagram (DFD)

A DFD is a graphical tool that shows the flow of data through an organization. It hides complexities and focuses on processes.

  • Data Flow (Arrow): A pipeline for data.
  • Data Store (Open-ended Rectangle): A repository for data (e.g., a database table).
  • Process (Oval or Rounded Rectangle): An action that transforms data.
  • External Entity (Rectangle): A source or destination of data outside the system.

2.11 & 2.12 DBA vs. DA

Data Administrator (DA): A managerial role. Responsible for the overall information resource, policies, and planning. (Non-technical)

Database Administrator (DBA): A technical role. Responsible for the physical DBMS, performance, security, and recovery. (Technical)

2.13 Data Dictionary

A repository of metadata that describes the logical structure of the database. It is used to manage data, communicate with users, and record changes.

  • Active Data Dictionary: Checked by the DBMS *every time* the database is accessed. Always consistent.
  • Passive Data Dictionary: Not used in day-to-day processing.

2.14 & 2.15 Logical vs. Physical Database Design

  • Logical Design: Creating the conceptual model (e.g., an E-R diagram). This involves representing entities, relationships, merging relations, and normalizing them.
  • Physical Design: Implementing the logical design on a specific DBMS. This involves:
    • Data Volume Analysis: Estimating the size of the database.
    • Data Distribution Strategy: Deciding where to store data (Centralized, Partitioned, Replicated).
    • File Organization: How records are arranged on disk (Sequential, Direct/Random, Indexed Sequential).
    • Indexes: Creating tables to speed up sorting and searching.

Chapter 3: The Entity-Relationship (E-R) Model

3.1 & 3.2 Elements of the E-R Model

The E-R model (by Peter Chen, 1976) is a visual tool for conceptual design. It's simple, provides a clear visual representation, and is an effective communication tool.

  • Entity: A person, place, thing, or event about which data is collected (e.g., STUDENT, BOOK).
    • Entity Type: The set of all entities with the same attributes (e.g., the STUDENT class).
    • Entity Instance: A single member of the set (e.g., the student 'Abdullah').
  • Attribute: A characteristic or property of an entity (e.g., Name, Address).
  • Relationship: A logical connection or association between entities (e.g., a STUDENT "Studies" a COURSE).

3.3.1 Types of Attributes

Simple: Cannot be subdivided (e.g., Gender).
Composite: Can be subdivided (e.g., Address -> Street, City).
Single-Valued: Has only one value (e.g., Age).
Multi-Valued: Can have many values (e.g., Skills).
Stored: Physically stored in the database (e.g., DateOfBirth).
Derived: Calculated from a stored attribute (e.g., Age is derived from DateOfBirth).
Identifier (Key): An attribute that uniquely identifies one instance (e.g., StudentID). Can be simple (one attribute) or composite (multiple attributes).

3.3.2 Types of Entities

  • Strong Entity: Can exist on its own and has its own unique identifier (e.g., EMPLOYEE). Represented by a single-line rectangle.
  • Weak Entity: Cannot exist without its "owner" strong entity. Its identifier is borrowed from the owner (e.g., DEPENDENT). Represented by a double-line rectangle.
  • Associative Entity: An entity that associates the instances of one or more entity types. Often used to "resolve" a many-to-many relationship and store attributes about that relationship.

3.4 Degree of Relationships

  • Unary (Recursive): A relationship between instances of the same entity type (e.g., a PERSON "Lives With" another PERSON).
  • Binary: A relationship between two entity types. This is the most common.
    • One-to-One (1:1): A CHAIRMAN "Runs" one DEPARTMENT.
    • One-to-Many (1:N): A COLLEGE "Has" many DEPARTMENTS.
    • Many-to-Many (M:N): A STUDENT "Studies" many COURSES.
  • Ternary: A relationship between three entity types (e.g., a CONSULTANT, CLIENT, and CONTRACT are all linked by a "Signs" relationship).

3.4.4 Cardinality Constraints

Specifies the number of instances of one entity that can be associated with an instance of another entity. It defines the "business rules."

  • Maximum Cardinality: The maximum number (1 or Many).
  • Minimum Cardinality: The minimum number (0 or 1).
    • If minimum is 0, participation is Optional.
    • If minimum is 1, participation is Mandatory.

3.5 Subtype & Supertype Entities

A way to model IS-A relationships (e.g., "A STUDENT IS-A PERSON").

  • Supertype: The general entity (e.g., PERSON) that contains common attributes (Name, Address).
  • Subtype: The specific entity (e.g., STUDENT) that contains specific attributes (RollNo) and inherits all attributes from the supertype.
  • Generalization: The process of identifying a more general entity type (bottom-up).
  • Specialization: The process of identifying more specific entity types (top-down).

Chapter 4: The Semantic Object Model

4.1 & 4.2 Semantic Object Model

An alternative to the E-R model that views the system as a collection of objects. It's considered more closely aligned with a user's perception of the world.

A Semantic Object is a "thing" in the user's environment, grouped into classes (e.g., STUDENT class). It has attributes:

  • Simple Attributes: Contain a single value.
  • Group Attributes: Composites of other attributes.
  • Semantic Object Attributes: An attribute that *is* another semantic object (this is the key difference from E-R).

Attribute Cardinality

Specified in n.m format, where 'n' is the minimum (0 or 1) and 'm' is the maximum (1 or N). E.g., 1.1 (mandatory one), 0.N (optional many).

4.3 Attribute Domains

A collection of all possible values for an attribute. A domain has two descriptions:

  • Physical Description: The data type (e.g., "string", "Roll No should not exceed 99").
  • Semantic Description: The function or purpose of the attribute (e.g., "Name of a student must be a valid name").

4.4 Semantic Object Views

A part of an object that is displayed in a particular application. One object can be displayed in many different views (e.g., one view for students, one for professors, both from the same COLLEGE object).

4.5 Types of Objects

  • Simple Object: Contains only single-valued, non-object attributes.
  • Composite Object: Contains one or more multi-valued, non-object attributes.
  • Compound Object: Contains at least one object attribute.
  • Hybrid Object: A combination of composite and compound.
  • Association Object: An object that links two or more other objects, typically storing data about that relationship.
  • Parent/Subtype Objects: A subtype object inherits the attributes of its parent (similar to E-R super/subtypes).
  • Archetype/Version Objects: An archetype object produces other objects that are "versions" of it (e.g., BOOK is the archetype, EDITION is the version).

Chapter 5: Relational Model & Normalization

5.1 Relational Model

The standard data model (by E.F. Codd, 1970) where data is stored in tables.

Terminology:

  • Relation: A table.
  • Tuple: A row (a single record).
  • Attribute: A column (a field).
  • Domain: The set of all possible values for an attribute.
  • Degree: The number of attributes (columns).
  • Cardinality: The number of tuples (rows).

5.2 Keys

Keys are attributes used to identify tuples (rows) in a relation.

  • Super Key: Any attribute or set of attributes that uniquely identifies a tuple.
  • Candidate Key: A minimal super key (no attribute can be removed and it still be unique).
  • Primary Key (PK): The one candidate key chosen by the designer to be the main identifier. It cannot contain null values.
  • Alternate Key: A candidate key that was *not* chosen to be the primary key.
  • Composite Key: A primary key that consists of two or more attributes.
  • Foreign Key (FK): An attribute in one relation whose values match the primary key in another relation, used to link the tables.

5.3 RDBMS Components

A Relational Database Management System (RDBMS) is the software that manages relational databases. Its components include a File Manager, Database Manager, Query Processor, and Data Dictionary.

5.4 Types of Relations

  • Base Tables: A table that physically exists in the database.
  • Query Results: The (virtual) table of results from a query.
  • Views: A virtual table based on a query, but stored in the database as an object. Views provide security and query simplicity.

5.6 Codd's Rules

A set of 12 rules (0-12) proposed by E.F. Codd to define what makes a DBMS truly "relational." Key rules include the Information Rule (all data in tables), Guaranteed Access (all data accessible by table/PK/attribute), and Systematic Treatment of Nulls.

5.7 Relational Data Integrity

  • Entity Integrity: Ensures the primary key can never be null.
  • Domain Integrity: Enforces that all values in a column are from its defined domain (e.g., "Grades" must be A, B, C, D, or F).
  • Referential Integrity: Ensures that a foreign key value must either be null or must match a valid, existing primary key value in the parent table. This prevents "orphan" records.

5.8 Database Languages

  • Data Definition Language (DDL): Used to define the database schema (e.g., CREATE TABLE, ALTER TABLE).
  • Data Manipulation Language (DML): Used to manipulate data within the schema (e.g., INSERT, UPDATE, DELETE, SELECT).

5.9 Relational Algebra

A procedural query language that defines operations on relations (tables) to produce new relations.

Unary Operations (one relation):

  • Selection (σ): Selects rows (tuples) that satisfy a condition. (Filters horizontally)
  • Projection (π): Selects columns (attributes). (Filters vertically)

Binary Operations (two relations):

  • Union (∪): Combines all rows from two compatible tables.
  • Difference (-): Returns rows that are in the first table but not the second.
  • Intersection (∩): Returns rows that appear in both tables.
  • Product (x): Creates a new table of all possible combinations of rows from two tables.
  • Division (÷): Finds rows in one table that match all rows in another.

5.10 Joins

A join operation combines data from two or more tables into a single result table based on matching criteria (the join condition).

  • Theta Join: A join that links tables based on any comparison operator (theta), such as =, <, >, or !=. It is a general-purpose join.
  • Equi Join: A type of Theta Join that only uses the equality operator (=) as its join condition. This is the most common type of join. The common attributes will appear twice in the result table.
  • Natural Join: A special type of Equi Join that joins tables on *all* attributes that have the same name. The duplicate columns are removed from the final result.
  • Outer Join: A join that retrieves all rows that match the join condition, *plus* all rows from one or both tables that do not have a match.
    • Left Outer Join: Includes all rows from the *left* table and matching rows from the right table (or NULLs if no match).
    • Right Outer Join: Includes all rows from the *right* table and matching rows from the left table (or NULLs if no match).
    • Full Outer Join: Includes all rows from *both* tables, filling with NULLs where matches are not found.
  • Semi Join: A join that returns only the rows from the first table for which a match is found in the second table. It does not include any columns from the second table.

5.13 Database Anomalies

Problems that occur in poorly designed (un-normalized) tables:

  • Insertion Anomaly: Inability to add data because other, unrelated data is missing.
  • Deletion Anomaly: Deleting one piece of data unintentionally deletes another, unrelated piece of data.
  • Modification Anomaly: A simple change (like an address) must be updated in many different rows, leading to inconsistency.

5.14 - 5.25 Normalization

The process of producing a simpler, more reliable database structure by breaking down complex tables to reduce redundancy and eliminate anomalies. This relies on the concept of Functional Dependency (if A determines B, then B is functionally dependent on A).

  • First Normal Form (1NF): All attributes must be atomic (no repeating groups or multi-valued attributes).
  • Second Normal Form (2NF): Must be in 1NF, AND all non-key attributes must depend on the entire primary key (no partial dependencies).
  • Third Normal Form (3NF): Must be in 2NF, AND no non-key attribute can depend on another non-key attribute (no transitive dependencies).
  • Boyce-Codd (BCNF): A stricter 3NF. Every determinant must be a candidate key.
  • Fourth Normal Form (4NF): Must be in BCNF, AND have no multi-valued dependencies.
  • Fifth (5NF) & Domain Key (DK/NF): Higher forms that deal with join dependencies and all constraints. 3NF is typically sufficient for most business applications.
